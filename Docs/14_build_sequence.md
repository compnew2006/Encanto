# تسلسل البناء التقني (Logical Build Sequence)

هذا التسلسل يحوّل مراحل المشروع إلى مسار عمل تقني تراكمي (Incremental). كل مرحلة تبني على ما قبلها، وتُنتج مخرجات قابلة للفحص (Verifiable) فورا قبل بدء المرحلة التالية.

التسلسل مصمم خصيصا لدعم بيئة (Svelte 5 / SvelteKit + Go / chi / pgx / sqlc).

## 1. التأسيس التقني (Base Foundation)
**الهدف:** بناء الهيكل الأساسي الذي سيحمل بقية المشروع.
- **التنفيذ (Backend):**
  - تهيئة مشروع `Go` وإعداد `chi` router.
  - تجهيز اتصال `PostgreSQL` عبر `pgx`.
  - تطبيق مخطط البيانات (DB Schema) وتوليد دوال قاعدة البيانات باستخدام `sqlc`.
- **التنفيذ (Frontend):**
  - تهيئة مشروع `SvelteKit` مع `Svelte 5` و Vite.
  - إعداد التوجيه الأساسي (Routing Skeleton) والمكونات الأساسية (Layouts).
- **التحقق المشروط:** يمكن تشغيل السيرفرين (Go و SvelteKit) دون أخطاء. وتعمل نقطة فحص (Health Check).

## 2. الهوية وسياق المستخدم (Auth & Context)
**الهدف:** تأمين الوصول ومنع الدخول العشوائي للمساحات التشغيلية.
- **التنفيذ:** 
  - `Backend`: بناء نقاط نهاية تسجيل الدخول وتوليد رموز الجلسة (JWT / Cookies).
  - `Frontend`: بناء صفحات Login، حفظ الجلسة، وصناعة الحواجز (Auth Guards) لحماية صفحات التطبيق.
- **التحقق المشروط:** لا يمكن الدخول لمسارات `/chat` بدون تسجيل دخول، وحالة تسجيل الخروج تنهي الجلسة بنجاح.

## 3. طبقة الصلاحيات وأساس البيانات (Permissions & Core Models)
**الهدف:** إخفاء ما لا يحق للمستخدم رؤيته وبناء النماذج الأساسية.
- **التنفيذ:**
  - `Backend`: إضافة Middlewares للتحقق من الأدوار (Role Checks). تطبيق فلاتر الرؤية (Visibility Scope) مباشرة في استعلامات `sqlc`.
  - `Frontend`: تعديل واجهة المستخدم لإخفاء الأزرار أو إبطالها (Disable) لمن لا يمتلك الصلاحية، وإنشاء شاشات الإعداد الأساسية.
- **التحقق المشروط:** مستخدم بصلاحية `read-only` يرى البيانات لكن يفشل عبر الـ API ويتعطل الزر عنده في الواجهة عند محاولة الحفظ.

## 4. مساحة عمل المحادثة (Conversation Workspace)
**الهدف:** الحصول على واجهة قابلة للقراءة والفلترة للمحادثات الحالية.
- **التنفيذ:**
  - `Backend`: إنشاء APIs لاستدعاء قائمة المحادثات (Inbox) والمحادثة الفردية مع الملاحظات وسجل الأحداث.
  - `Frontend`: تصميم وبناء صفحة `/chat` و `/chat/[contactId]` بصيغتها الجامدة (دون تزامن حي بعد).
- **التحقق المشروط:** يمكن رؤية المحادثات الموجودة مسبقاً في قاعدة البيانات على الشاشة، ويمكن فتح أي محادثة ورؤية تفاصيلها.

## 5. الإرسال والاستقبال وقاعدة الأحداث (Sending & Event Logistics)
**الهدف:** تفعيل الاتجاهين للإرسال.
- **التنفيذ:**
  - `Backend`: بناء واجهة إرسال الرسالة (Message Ingestion) وضخ رسالة الصادر إلى قاعدة البيانات. تجهيز نمط الـ Outbox.
  - `Frontend`: تفعيل مربع الإرسال (Composer) وأزرار الـ (Assign/Close).
- **التحقق المشروط:** عند إرسال المحادثة تظهر مباشرة كسطر جديد في قاعدة البيانات (سلة الرسائل)، دون الحاجة لتحديث الصفحة المباشر مؤقتاً.

## 6. الوقت الفعلي والإشعارات (Realtime & WebSockets)
**الهدف:** تحريك الواجهة الجامدة وجعلها حية بناء على أحداث النظام.
- **التنفيذ:**
  - `Backend`: بناء خادم الـ `WebSocket` داخل `Go`. استخدام `Redis` كموزع للأحداث (Pub/Sub) لتوصيل التغييرات بناءً على الـ Outbox.
  - `Frontend`: إنشاء عميل WS داخل مخازن السفلتي (Svelte Stores) ليستقبل الإشارات ويرسمها فورا في قائمة المحادثات ولائحة الإشعارات.
- **التحقق المشروط:** إدخال رسالة وهمية لـ DB مباشرة يتم ترجمتها كرسالة بصرية فورية في الواجهة دون عمل `Refresh`.

## 7. الحسابات التشغيلية والربط الفعلي (Account & WhatsApp Operations)
**الهدف:** جعل صندوق الوارد مرتبطا بمصدر حقيقي.
- **التنفيذ (Phase 2):** 
  - `Backend`: ربط `whatsmeow` بمستقبلات الأحداث الواردة والصادرة.
  - `Frontend`: شاشة `Instances` لعرض وتشغيل ومسح الـ QR Code لربط الأرقام.
- **التحقق المشروط:** إرسال رسالة واتساب حقيقية للرقم الافتراضي تصل وتظهر حية على المنصة وباسم الجهة المرسلة.

## 8. الإعدادات، جهات الاتصال، والمحادثات المغلقة
**الهدف:** استكمال السطح الإداري المساعد.
- **التنفيذ:**
  - واجهات وجهات اتصال (CRUD Contacts).
  - فلاتر وتصفح المحادثات السابقة (Closed Chats) مع خيار إعادة الفتح (Reopen) والذي يبث إشعار Realtime.
  - واجهات إعداد المنظمة.
- **التحقق المشروط:** يمكن تعديل معلومات جهة، إضافتها للإعدادات، ورؤية التأثير في شاشة الـ Chat.

## 9. الترخيص والحدود الموثوقة (Licensing & Quotas)
**الهدف:** حماية المنصة من الاستنزاف.
- **التنفيذ:**
  - `Backend`: إضافة مستشعرات الحمولات (Quotas middlewares) ومسارات تفعيل الرخص.
  - `Frontend`: صفحة `License` وإعادة توجيه المستخدم لصفحة الـ `cleanup` عند كسر حاجز الموارد.
- **التحقق المشروط:** تجاوز الحد المبرمج مسبقاً (مثلا أكثر من 5 مستخدمين) يؤدي لرفض طلبات الـ HTTP من قبل الـ API.

## 10. التحليلات، والحملات، ومهام الخلفية (Analytics, Campaigns, Background Jobs)
**الهدف:** أتمتة العمل والمراقبة.
- **التنفيذ:**
  - `Backend`: مجدول مهام (Cron) باستخدام Go و Redis لقوائم الحملات وتنقيح الملفات القديمة.
  - تجميع البيانات باستخدام `sqlc` لعرض Analytics.
  - `Frontend`: استهلاك الواجهات ورسم جداول/رسومات التحليلات والحملات.
- **التحقق المشروط:** إطلاق حملة مسجلة تجدول العمليات في الـ `job_runs` وتعطي إشارات تقدم.

## 11. التدقيق، الاطمئنان والتسليم (Verify & Handoff)
**الهدف:** إنهاء وضخ المنتج بنسخته الصلبة الأولى (V1).
  - تشغيل كافّة اختبارات الـ (End-to-End) عبر `Playwright` للتأكد من عدم كسر أي ميزة.
  - مراجعة الـ Audit Logs للتأكد من تسجيل الحركات المفصلية الحساسة.
- **التحقق المشروط:** اجتياز كامل للاختبارات + توفر بيئة جاهزة للإطلاق التجريبي (Staging).
